# Propagation терминологических изменений

Когда перевод термина меняется (на любом этапе pipeline), изменение должно быть атомарно пропагировано во все затронутые файлы. Расхождений между dictionary и переведёнными файлами быть не должно.

## Триггеры

Изменение перевода термина происходит когда:
- Верификатор (translate-verify, verify-content) предлагает другой перевод термина
- Fix-цикл в translate-content меняет перевод термина
- Пользователь вручную меняет термин
- Кросс-версионная проверка обнаруживает конфликт

## Процедура изменения термина

### Шаг 1: Обнаружение и подсчёт

1. Зафиксировать: `{EN термин}`, `{старый RU перевод}`, `{новый RU перевод}`
2. Grep по EN файлам: подсчитать количество вхождений `{EN термин}`
3. Grep по RU файлам: подсчитать количество вхождений `{старый RU перевод}`
4. Сопоставить: EN count и RU count должны быть сопоставимы (с допуском на падежные формы)

### Шаг 2: Классификация изменения

**Критическое изменение** (требует ручного подтверждения):
- Термин используется в 3+ файлах
- Конфликт с записью в `logs/`
- Конфликт с общим словарём `src/translate/`
- Изменение базового термина из `01_dictionary_base.md`

**Обычное изменение** (автоматическое):
- Термин в 1-2 файлах
- Нет конфликтов с логом или общим словарём
- Entity-термин (заклинание, монстр, предмет)

Критические изменения → AskUserQuestion перед применением.

### Шаг 3: Применение

1. **Обновить dictionary:** найти термин в `src/{game}/translate/*_dictionary_*.md`, заменить перевод
2. **Обновить RU файлы:** заменить `{старый RU перевод}` → `{новый RU перевод}` во всех RU файлах (с учётом падежных форм)
3. **Обновить `*_Glossary/`:** если термин есть в глоссарии версии — обновить там тоже
4. **Записать в лог:** `src/{game}/translate/logs/{date}_log.md`

### Шаг 4: Верификация пропагации

После замены — контрольная проверка:

1. Grep `{старый RU перевод}` по всем RU файлам → должно быть 0 вхождений
2. Grep `{новый RU перевод}` по всем RU файлам → count должен совпадать с EN count
3. Проверить dictionary: термин обновлён
4. Если count не совпадает → ручная проверка (возможно, падежные формы не все заменены)

```
Propagation: "{EN}" "{старый RU}" → "{новый RU}"
  dictionary: обновлён ✓
  RU файлов затронуто: N
  EN вхождений: X, RU вхождений после замены: Y
  Остаток старого перевода: 0 ✓
```

## Инвариант: dictionary ↔ logs

- **dictionary** — текущее состояние (что переводить как)
- **logs** — история изменений (когда и почему изменилось)
- Каждый `revision` в логе **обязан** иметь соответствующее обновление в dictionary
- Каждое изменение в dictionary **обязано** иметь запись в логе (кроме первичного создания словаря)

### Валидация инварианта

При верификации (`/verify-content`, `/translate-verify`):

1. Прочитать все записи типа `revision` из `logs/`
2. Для каждой: проверить что dictionary содержит актуальный перевод
3. Если dictionary содержит старый перевод → **error**: лог и словарь рассогласованы

## Conflict resolution

Единое правило разрешения конфликтов:

**Более поздняя явная запись в логе побеждает любой словарь. Системный словарь побеждает общий, если нет записи в логе.**

Приоритет (от высшего к низшему):
1. `src/{game}/translate/logs/` — последняя запись по дате (абсолютный приоритет)
2. `src/translate/logs/` — общие решения
3. `src/{game}/translate/01_dictionary_base.md` — системный словарь
4. `src/translate/01_dictionary_base.md` — общий словарь
5. `src/{game}/translate/02+_dictionary_*.md` — словари сущностей
6. `src/{game}/{version}/ru/*_Glossary/` — глоссарий версии (производный)

## Verify loop: типизация ошибок

Цикл верификации завершается при:
- **0 errors** (критические ошибки, пропущенный контент, терминологические несоответствия)
- **warnings** — каждый явно принят (accepted) или отложен (deferred) с обоснованием
- **notes** — не блокируют завершение

Типы ошибок — см. `.claude/rules/translation-validation.md`, раздел 8.
