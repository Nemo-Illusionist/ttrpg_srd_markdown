# /cleanup-artifacts — Сведение, разбивка и чистка

## Использование

```
/cleanup-artifacts <game> <version>
```

Пример: `/cleanup-artifacts brp srd-1.0`

## Предварительные условия

Файлы из `/convert-pdf` должны существовать в `/tmp/`:
- `/tmp/{game}_marker.md`
- `/tmp/{game}_pymupdf.md`
- `/tmp/{game}_docling.md`

Также нужен оригинальный PDF для сверки (спросить путь если неизвестен).

## Алгоритм

### Фаза A: Сведение (merge)

Создай **Agent** (обязательно **model: "opus"**) для сведения.

Агент анализирует все три результата конвертации + оригинальный PDF:

1. **Определи структуру документа:**
   - Прочитай все три файла (чанками если большие)
   - Определи главы и разделы по заголовкам
   - Сверь с оглавлением PDF (если есть)

2. **Бери лучшие части из каждого конвертера:**
   - **Заголовки** — обычно точнее у marker (правильные уровни `#`)
   - **Таблицы** — обычно лучше у docling (правильная структура), сверяй с marker
   - **Текст и списки** — сравнивай все три, выбирай наиболее точный
   - **При расхождениях** — сверяй с PDF (через Read tool для PDF)

3. **Собери единый markdown** — `/tmp/{game}_merged.md`

4. **Отчёт о полезности конвертеров:**

После сведения выведи статистику:

```
Полезность конвертеров для {game} {version}:

marker:
  - Использовано для: N% текста, N% таблиц, N% заголовков
  - Сильные стороны: ...
  - Слабые стороны: ...

pymupdf4llm:
  - Использовано для: N% текста, N% таблиц, N% заголовков
  - Сильные стороны: ...
  - Слабые стороны: ...

docling:
  - Использовано для: N% текста, N% таблиц, N% заголовков
  - Сильные стороны: ...
  - Слабые стороны: ...

Итого: {основной источник} — основной, {второй} — для таблиц, {третий} — верификация
```

Проценты рассчитывай по количеству строк, взятых из каждого конвертера.

### Фаза B: Разбивка на файлы

1. **Определи точки разбивки** по структуре документа:
   - Каждая глава/крупный раздел → отдельный файл
   - `00_Legal.md` — лицензия/OGL всегда первым файлом
   - Нумерация: `00_Legal.md`, `01_Name.md`, `02_Name.md`, ...

2. **Именование по паттерну проекта:**
   - Путь: `src/{game}/{version}/en/NN_Name.md`
   - Формат номера: двузначный с ведущим нулём
   - Имя: CamelCase из названия главы (пробелы → без пробелов)
   - Если глава содержит подтипы (классы, расы и т.д.) → подпапка `NN_Classes/01_Fighter.md`

3. **Глоссарий НЕ создаётся** (это задача `/build-glossary`)

4. **Коммит после каждого файла:**
   ```
   Импорт {game} {version}: {filename}
   ```

### Фаза C: Чистка артефактов

Чеклист: **`.claude/rules/pdf-cleanup.md`** — единый источник правды.

Агент ОБЯЗАН прочитать этот файл и пройтись по каждому пункту для каждого созданного файла.

**Коммит после чистки каждого файла:**
```
Импорт {game} {version}: чистка {filename}
```

### Финальная сводка

```
Сведение и чистка завершены:

Файлы созданы: K
- src/{game}/{version}/en/00_Legal.md
- src/{game}/{version}/en/01_....md
- ...

Артефактов исправлено: M
- Разрывов строк: ...
- Номеров страниц: ...
- Сломанных таблиц: ...
- Проблем форматирования: ...

Следующий шаг: /verify-import {game} {version}
```

## Технические требования

- Агент сведения — **model: "opus"** (обязательно)
- Большие файлы (>3000 строк) — чтение чанками через offset/limit
- Коммит после каждого файла (не групповой)
- Сообщения коммитов на русском
- Оригинальный PDF нужен для сверки при расхождениях
